## 一、问题：当前AI上下文与记忆的7大劣势

### 1.1 容量天花板 —— memory 工具 2200 字符硬上限

这不是理论问题，是我们每天面对的实操瓶颈。Hermes Agent 内置 memory 系统只有~2200字符，写入超限后新条目被静默截断，没有任何警告。结构化数据（JSON/YAML/配置列表）根本无法完整存入。每次接近上限就得手工删旧条目——代价是丢失之前沉淀的信息。

**实操表现**：本报告撰写前，memory 已填满 2,192/2,200，为加一条 GitHub 仓库信息不得不再做一次"替换旧条目"操作。

### 1.2 跨会话记忆断裂 —— session_search 是搜索，不是记忆

session_search 本质是 FTS5 全文搜索，不是持续记忆。每次跨会话查找都依赖关键词匹配——默认 AND 语义导致多关键词查询几乎必返回空，必须手工改用 OR。用户频繁感觉"我应该知道这个"但系统确实不记得。每次新会话从空白启动，昨天刚讨论完的话题今天必须从头加载。**没有优先级标记机制**来区分哪些是核心记忆、哪些是临时缓存。

### 1.3 上下文窗口溢出 —— 无声驱逐

对话变长后，旧消息被静默丢弃。系统不会告诉你"你之前的3轮对话已经丢失"。早期做出的决策依据、分析过程在溢出后消失，导致后续回答与前期结论矛盾。**更致命的是：系统不知道自己丢失了什么**。

### 1.4 多任务混合 —— 上下文污染

同一会话中先后处理多个不相关任务时，前一个任务的上下文残留污染后一个任务的推理。历史工具调用结果（文件列表、搜索返回、代码片段）占用宝贵上下文空间但对当前任务毫无价值。系统有时会错误地将前一个任务的上下文元素应用到当前问题中，产生幻觉。**没有会话分支/隔离机制**。

### 1.5 技能加载的盲目性

系统不知道当前任务应该加载什么技能，完全依赖用户指令或预设。即使之前证明某个技能非常有用，如果在当前会话中未被显式激活，系统也不会主动调用。多个技能同时加载时可能冲突。**没有技能卸载机制**——不需要的技能持续占用空间。

### 1.6 Token消耗与上下文长度的矛盾

越用越贵。对话越长，每次调用的 Token 消耗越大。工具调用记录膨胀——每次文件操作、搜索结果都完整保留在上下文中，快速填满窗口。**没有预算机制**来区分"必须保留"和"可以压缩归档"的上下文。

### 1.7 单Agent信息孤岛

当前 session 的所有记忆、上下文、中间结果全部对其他 session 不可见。无法同时从多个视角看同一个问题（写代码+审查）。每次解决问题学到的经验在会话结束后就消失了。一切依赖当前会话的完整性——一旦受损，全盘偏离。

## 二、诺亚核心：系统方案

针对上述7大劣势，诺亚核心构建了一套多层、有组织、自管理的记忆与上下文系统：

### 2.1 双层架构：memory + vault

```
┌────────────────────────────────────────────────┐
│              memory 工具（~2200字符）            │
│  会话级关键事实 + 索引指针（指向 vault 位置）     │
│  角色：高频缓存 + 导航                            │
└──────────────────────────────────────────┬─────┘
                                           │ 只有指针
                                           ▼
┌────────────────────────────────────────────────┐
│              vault/ 永久仓库（磁盘级）            │
│  ├─ credentials/   密钥配置                      │
│  ├─ configs/       服务器/工具配置              │
│  ├─ team/          活跃角色身份卡（13个）        │
│  ├─ 先贤祠/        归档角色墓志铭（8个）         │
│  ├─ projects/      永久项目档案（28项）          │
│  ├─ 公共技能/       跨角色可复用技能包           │
│  └─ index.md       总索引                        │
└────────────────────────────────────────────────┘
```

**解决问题：** 1.1（容量天花板）—— 2200字符存指针，全量文档进 vault，容量从~2200扩展到磁盘级别。

### 2.2 温度分层：生命周期管理

| 层级 | 冷却期 | 存储粒度 | 对应传统上下文的问题 |
|------|--------|---------|-------------------|
| 🔥 热记忆 | <7天 | 完整文件 | 替代"全部保留"的粗暴策略 |
| 🌤 普通记忆 | 7-30天 | 完整文件 | 最近访问的项目自然降温 |
| ❄️ 冷记忆 | 30-90天 | 压缩摘要 | 旧但可能有价值的项目 |
| 🏚 仓库记忆 | >90天 | 一句话指针 | 永久但罕见的记忆 |

**解决问题：** 1.3（窗口溢出）+ 1.6（Token消耗矛盾）—— 不是所有历史都要留在上下文中。温度分层让活跃内容保持完整，冷却内容自然降级为摘要。

### 2.3 四区块分类：角色/项目/业务技能/普通

每个区块有独立的温度索引和存储目录，互不干扰。

**解决问题：** 1.4（多任务污染）—— 不同维度的记忆物理隔离，项目记忆不会污染角色记忆。

### 2.4 三卡一体：角色身份卡体系

| 模块 | 内容 | 生命周期 |
|------|------|---------|
| **永久留存** | 姓名、灵魂、核心性格、精简履历 | 永不删除 → 先贤祠 |
| **动态操作**（软日志） | 每日操作留痕 | 归档时清理 |
| **能力记忆**（技能包） | 可复用知识/流程 | 按热度升级流转 |

**解决问题：** 1.7（信息孤岛）—— 角色身份不因会话结束而丢失。技能包在团队内按热度流转。

### 2.5 项目6文件模板：结构化追踪

每个项目7个标准化文件，进行中维护5个动态文件，归档时补2个终版文件。

**解决问题：** 1.2（跨会话断裂）+ 1.7（经验丢失）—— 项目经验结构化入库，而非依赖不可靠的搜索。

### 2.6 7条铁律：规则即代码

| # | 铁律 | 解决什么 |
|:-:|:-----|:---------|
| 1 | 温度索引必须更新 | 无自动更新=系统失效 |
| 2 | 动态文件不等结束 | 经验不及时记=丢失 |
| 3 | 归档文件不提前写 | GCat确认前乱动=回退 |
| 4 | 不确定就问GCat | 猜测=错误归档 |
| 5 | 规则落地补丁 | 规则只留在设计文档=无人执行 |
| 6 | 文件删除检查 | 误删无备份=永久丢失 |
| 7 | 自指闭环 | 系统特权=规则污染 |

**解决问题：** 1.5（技能加载盲目性）—— 铁律固化执行路径，不依赖"猜该做什么"。

### 2.7 任务书系统：执行信号明确化

任务书 = GCat 的正式执行指令。任务按类型分流：开发/调研/运维/设计/写作/其他，每类型调对应技能。

**解决问题：** 1.4（多任务混合）—— 任务物理隔离在独立文件中，类型决定技能选择。

### 2.8 对话扫描机制：实时更新

每10-15轮对话自动扫描，匹配项目则追加记录，无匹配丢弃，模糊归属问GCat。

**解决问题：** 1.2（跨会话断裂）—— 对话中实时将结构化信息写入持久化文件，不等会话结束。

## 五、核心价值主张

诺亚核心解决的是AI Agent的**结构性失忆症**——每个会话结束后AI回到起点，经验不积累、知识不沉淀、规则不延续。

- ✅ 是一套 **AI自身的记忆管理方法论**，温度分层为生命周期
- ✅ 是一个 **执行纪律系统**，7条铁律确保规范
- ✅ 是一个 **Token预算系统**，热度决定存储粒度
- ✅ 是一个 **文明传承系统**，角色可归档、经验可流转

**一句话：诺亚核心让AI Agent记住"该怎么记住"——不是更大容量的记忆，而是有纪律、有结构、有生命周期的记忆管理。**
